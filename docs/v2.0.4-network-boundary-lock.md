# v2.0.4 Wizard Networking Boundary Lock

## Purpose

Define the first active networking boundary for `uDOS-wizard` in `v2.0.4`
without leaking provider routing, secret ownership, or remote bridge logic into
`uDOS-shell`, `uDOS-core`, or sibling repos.

## Boundary Rules

- `uDOS-wizard` owns provider routing, remote bridge calls, and any secret-backed
  network lanes
- `uDOS-shell` may call Wizard-owned HTTP surfaces, but it does not own provider
  adapters, secrets, or remote networking policy
- `uDOS-core` remains contract-first and does not absorb Wizard networking logic
- `uHOME-server` and `uHOME-empire` consume Wizard network contracts rather than
  defining competing bridge semantics

## First Live Handoff

The first live shell-to-Wizard boundary landed as a narrow assist handoff:

- shell command: `#wizard assist topic:<task>`
- shell transport: HTTP GET to Wizard `/assist`
- config surface: `UDOS_WIZARD_HOST`, `UDOS_WIZARD_PORT`
- default base URL: `http://127.0.0.1:8787`
- shell fallback: if Wizard is unavailable, keep rendering the preview envelope
  and surface the error instead of silently dropping the route

This is intentionally narrow. It proves the boundary without yet committing to
the broader remote provider and secret-backed bridge surface.

## Not In Scope Yet

- MCP ↔ VS Code integration
- provider-specific remote bridges
- secret-backed network credentials beyond Wizard-owned storage
- Shell-owned retry, bridge, or provider policy

These remain inside the active `#binder/wizard-v2-0-4-network-boundaries`
sequence until the ownership boundary is fully locked.

## Next Steps

1. Document the public Wizard route set Shell may call directly
2. Define the secret-backed bridge lanes shared with `uHOME-server` and `uHOME-empire`
3. Expand from assist-only handoff to workflow/orchestration handoff where justified
4. Only then expose the MCP registry as a VS Code-consumable MCP server
