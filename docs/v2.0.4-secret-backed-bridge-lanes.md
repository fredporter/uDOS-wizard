# v2.0.4 Secret-Backed Bridge Lanes

## Purpose

Define the first secret-backed bridge lanes shared between Wizard,
`uHOME-server`, and `uHOME-empire` so the family does not duplicate secret
storage or network ownership.

## Family Rule

- Wizard owns secret-backed credentials and remote provider routing
- sibling repos may contribute payloads, execution context, and local services
- sibling repos do not become the canonical owner of provider credentials,
  webhook secrets, or remote network bridge policy

Wizard now publishes the first household networking policy artifact for sibling
consumption:

- contract: `contracts/uhome-network-policy-contract.json`
- schema: `contracts/uhome-network-policy.schema.json`
- route: `GET /contracts/uhome/network-policy`
- schema route: `GET /contracts/uhome/network-policy/schema`
- validation route: `POST /contracts/uhome/network-policy/validate`

This artifact carries profile definitions for `Beacon`, `Crypt`, `Tomb`, and
`Home` without exposing raw secret values.

## `uHOME-server` Lane

`uHOME-server` owns local durable fulfillment.

Wizard-facing responsibilities:

- receive workflow automation handoffs from Wizard
- run local durable automation jobs behind Wizard-owned workflow policy
- return durable job results back to Wizard workflow reconciliation routes

Wizard-owned secret-backed pieces:

- remote provider credentials needed to trigger or authorize bridge work
- any shared bridge token or webhook signing secret used outside the local server
- family-wide retry, approval, and provider-routing policy

Server-owned non-secret pieces:

- local runtime config for the server itself
- durable job queue, results, and local status routes
- local network discovery and device/runtime fulfillment

## `uHOME-empire` Lane

`uHOME-empire` owns outbound ops payloads and sync containers.

Wizard-facing responsibilities:

- request workflow plans for provider-backed sync or publishing lanes
- call Wizard assist/orchestration surfaces where provider routing is required
- supply non-secret payloads and sync records to Wizard-owned provider bridges

Wizard-owned secret-backed pieces:

- provider API keys, OAuth tokens, webhook secrets, and bridge credentials
- outbound provider retry/backoff policy
- family-wide routing decisions for provider-backed sync or publishing

Empire-owned non-secret pieces:

- sync records, container manifests, pack payloads, and outbound templates
- local packaging and execution surfaces
- CRM/contact and publishing data models that do not contain shared secrets

## Not Allowed

- `uHOME-server` or `uHOME-empire` storing family-wide provider secrets as a
  competing source of truth
- Shell bypassing Wizard to talk directly to provider-backed bridge lanes
- Core owning bridge credentials, provider adapters, or remote retry policy

## Next Step

The next implementation lane is to document which Wizard routes back these
server and empire bridge calls and then expand from assist-only shell traffic to
workflow/orchestration traffic against the same ownership rules.
