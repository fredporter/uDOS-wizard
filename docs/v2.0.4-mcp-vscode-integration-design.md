# v2.0.4 MCP VS Code Integration Design

Status: historical design baseline with active implementation follow-through
Owner: uDOS-wizard
Tag: @dev/wizard-mcp-vscode
Binder: #binder/wizard-v2-0-4-network-boundaries

## Purpose

Define a boundary-safe design for exposing Wizard-managed MCP tooling to editor
agents after networking ownership is locked.

This document now serves as the baseline for the implemented local editor
client lane. The VS Code client and Wizard-managed MCP bridge exist; the value
here is the boundary framing and request envelope, not a claim that the runtime
path is still hypothetical.

## Current Baseline

Current Wizard MCP-related surfaces:

- GET /mcp/tools
- GET /ok/mcp-policy
- POST /ok/route
- wizard/mcp_registry.py
- wizard/ok/mcp_policy.py
- wizard/ok/routing_engine.py

The MCP registry is currently a bounded in-process registry with working local
discovery and invocation. Full external MCP server federation remains a future
expansion lane rather than the current contract.

## Boundary Guardrails

- Wizard remains the managed MCP owner for networked tool calls
- Core remains the contract and deterministic validation owner
- Shell and editor clients consume Wizard surfaces but do not own routing,
  secret storage, budget policy, or scheduling policy

## Proposed Integration Shape

Phase 1 (read-only editor visibility):

- expose tool list and policy summary through stable MCP-friendly adapters
- allow editor agents to discover Wizard-managed tools and policy constraints

Phase 2 (managed invocation gateway):

- route editor-originated tool requests through Wizard policy checks
- enforce budget, approval, and scheduling before invocation
- return routed, deferred, or rejected outcomes with audit fields

Phase 3 (deferred and scheduled execution):

- allow editor-originated requests to enter Wizard deferred lanes when blocked
- support explicit status inspection from editor surfaces

## Required Request Envelope

Editor-initiated requests should include at minimum:

- request_id
- project_id
- task_class
- allowed_budget_groups
- approval_granted
- schedule_class
- offline_sufficient
- cache_hit

## Required Decision Envelope

Wizard responses should include:

- status (routed, local, cache-hit, deferred, error)
- provider_id or route target
- task_class
- complexity
- budget_group
- schedule_class when deferred
- attempted providers with rejection reasons
- audit_required marker

## Out Of Scope In Round A

- direct editor access to provider credentials
- bypassing Wizard policy checks
- moving MCP bridge ownership into Shell, Core, or editor plugins
- unrestricted autonomous loops from editor clients

## Historical Exit Condition For Implementation Start

Implementation can begin only after the networking boundary lock remains
explicit and accepted in the active binder.
