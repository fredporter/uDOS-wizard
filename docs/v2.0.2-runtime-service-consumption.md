# v2.0.2 Runtime Service Consumption

`uDOS-wizard` consumes the `uDOS-core` runtime-service catalog to explain how
its orchestration surfaces attach to the shared platform contracts.

For packaged or non-sibling consumers, Wizard also publishes
`contracts/orchestration-contract.json` as the machine-readable export for its
shared orchestration routes and response shapes.

## Status Extensions

`/orchestration/status` now reports:

- `foundation_version`
- `runtime_services`
- `runtime_service_source`
- `orchestration_contract_source`
- `result_store_path`
- `result_store_mode`

The shared dispatch path is:

- `/orchestration/dispatch?task=<task>&mode=<mode>&surface=<surface>`
- `/orchestration/workflow-plan?objective=<objective>&mode=<mode>`

Dispatch responses now expose a common contract:

- `dispatch_version`
- `dispatch_id`
- `request`
- `route_contract`
- `callback_contract`

Result reporting paths are:

- `POST /orchestration/callback`
- `GET /orchestration/result/<dispatch_id>`

Wizard result reports now persist to a file-backed JSON store under
`memory/orchestration-results.json` so callback outputs survive registry
reloads during the `v2.0.2` release pass.

## Packaging Rule

- `uDOS-wizard` owns the orchestration route package at
  `contracts/orchestration-contract.json`
- sibling and non-sibling consumers should read that artifact when they need a
  stable exported route and contract surface
- live route responses may include contract hints, but the packaged artifact is
  the canonical export for release use

## Consumption Rule

- `uDOS-core` owns the runtime-service contract definitions
- `uDOS-wizard` reports which of those services it consumes from
  `uDOS-core/contracts/runtime-services.json`
- orchestration status may describe usage, but it does not replace the Core
  manifest
