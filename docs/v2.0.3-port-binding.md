# v2.0.3 Port Binding

The archived v1 Wizard port manager covered a much larger surface:

- multi-service registry
- conflict healing
- kill and reassign flows
- launch coordination across several dev services

Current v2 Wizard does not need that whole subsystem yet.

## Active v2 Rule

For the current application, Wizard needs a slim binding contract:

- choose host from `UDOS_WIZARD_HOST` or default `127.0.0.1`
- choose port from `UDOS_WIZARD_PORT` or default `8787`
- if the default port is busy, auto-shift to the next free port by default
- if an explicit port is requested and busy, fail with a clear occupant message

## Why This Split

The current repo only launches one primary service process.

That means the useful v1 concepts to retain are:

- conflict detection
- occupant reporting
- deterministic env-var override
- predictable startup behavior

The full v1 service registry should only come back if v2 Wizard again owns a
multi-service launcher surface.

## Environment Variables

- `UDOS_WIZARD_HOST`
- `UDOS_WIZARD_PORT`
- `UDOS_WIZARD_PORT_AUTO_SHIFT`

## Current Behavior

- default launch path: `.venv/bin/python -m wizard.main`
- if `8787` is free, Wizard uses it
- if `8787` is occupied, Wizard tries the next free port
- if `UDOS_WIZARD_PORT` is set and occupied, Wizard exits with a clear error
- Wizard exposes `GET /port/status` with the current bind snapshot or the
  configured desired bind when the actual bind is not known to the process
- startup prints the resolved GUI and Thin GUI URLs

This is the right level of port handling for the current launch surface.
